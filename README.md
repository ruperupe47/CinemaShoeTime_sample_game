## プロジェクトログライン
* プレイヤーは異なる特性を持つマシン（エンペラー、カイザー、ミカド、マシン）を操作し、迫り来るモノノケの軍勢を倒す 2D 見下ろし型・無双系アクションゲーム。

---

## プロジェクト概要
* **開発背景**: 研究室内において、「ここから一年かけて本開発を行うゲームプロジェクト」の先行プロトタイプとして企画・制作。
* **目的**: 2Dアクションにおける「大量のオブジェクトの効率的な描画・演算処理」、「自作の動的経路探索AIによる群衆制御」、「音声合成を用いた動的な状況変化演出」の実装と、ゲーム性の検証。
* **起動**: ゲームの起動にはmain.pyを使用。
* **想定環境**: コアライブラリの設計に準じ、現在は主に「macOS」環境での動作を想定しています。

---

## 開発中に起きた課題点

### 1. マルチスレッド化による処理の競合とデータの不整合
* **課題**: 敵のAI思考・移動処理やエフェクト演出の描画をマルチスレッドで実装していたが、敵の動作が不安定になったり、データが消失する不具合が発生した。
* **原因と対策**: スレッドライブラリで複数の敵・エフェクトを同時に呼び出していたことが原因。Cinema_ShoeTimeライブラリに新しくプレイヤー、敵、エフェクトをオブジェクトとして扱うためのクラスと関数を追加した。結果、ライブラリ内のメインループ内で全てのオブジェクトを1つ1つ同期処理する形にしたことで処理負荷と変数の競合によるデッドロックの問題を解決した。


### 2. 大量オブジェクト同時描画によるシステム負荷
* **課題**: 敵・攻撃エフェクトを同時に「毎フレームすべて」描画しようとしたため、描画負荷が発生し、フレームレートが著しく低下した。
* **原因と対策**: 画面外にあるオブジェクトまで愚直に描画APIを呼び出していたことが原因。対策として、プレイヤーを中心とした画面の有効描画範囲を計算する関数を実装し、そこから算出された相対座標を基準に「画面内に入っているオブジェクトのみを描画する」ように描画ロジックを最適化した。


### 3. プレイヤーとマップの座標更新タイミングのズレによる不自然なスクロールアニメーション
* **課題**: プレイヤーの移動アニメーションと、背景のスクロールアニメーションが同時に動いた際、キャラクターや背景が一瞬細かくガクつく、または位置がズレるような不自然な挙動が発生した。
* **原因と対策**: プレイヤーの座標更新のタイミングと、カメラの描画基準点を更新するタイミング、および各エネミー・背景マテリアルのアニメーションフレームの更新タイミングがメインループ内で同期していなかったことが原因。移動処理 ➔ カメラ位置確定 ➔ マップ・全オブジェクトの相対座標計算 ➔ 一括描画 という「1フレーム内での厳密な処理順序の固定」を行うことで、滑らかなスクロールアニメーションを実現した。
